# 변경 이력

## 1.0.0-next.1

`1.0.0-next.0`에서 도입한 인터랙티브 뒤로가기 제스처의 문제 세 가지를 수정했습니다. 모두 로컬 개발 환경보다는 실제 기기와 느린 네트워크에서 드러나는 문제여서 첫 프리릴리스 전에 발견되지 않았습니다.

### 무엇이 달라졌나

인터랙티브 제스처는 완료 시점을 예측할 수 없는 두 가지를 맞물려 진행해야 합니다. 하나는 브라우저의 View Transition 스냅샷이고, 다른 하나는 Next.js가 새 라우트를 커밋하는 것(RSC 페치)입니다. 이전 구현은 라우트 커밋이 빠르게 끝난다고 가정하고 있었고, 아래 수정 사항은 모두 그 가정이 깨지는 지점입니다.

라우트가 끝내 커밋되지 않는 상황을 대비한 안전장치 타임아웃이 `duration + 400ms`(기본 680ms)로 잡혀 있었습니다. 네트워크 왕복 시간이 아니라 애니메이션 길이에서 끌어온 값입니다. prefetch가 빗나가거나 연결이 느리면 라우트 커밋이 이 시간을 넘겼고, 그러면 타임아웃이 DOM에 여전히 이전 화면이 떠 있는 상태로 전환을 *확정*해버렸습니다. 브라우저는 서로 동일한 두 스냅샷을 캡처해 아무 의미 없는 전환을 재생했고, 실제 화면 교체는 전환이 끝난 뒤 애니메이션 없이 일어났습니다. 사용자 눈에는 화면이 툭 하고 건너뛰는 것처럼 보였습니다. 이제 타임아웃은 네트워크 왕복 시간을 기준으로 잡히며, 만료되면 전환을 확정하는 대신 포기합니다. 이동 자체는 그대로 이뤄지고 전환 상태도 중간에 걸린 채 남지 않고 정리됩니다.

### 수정

- **라우트 커밋이 느리면 화면이 애니메이션 없이 교체되던 문제.** 안전장치 타임아웃을 애니메이션 길이에서 파생시키지 않고 네트워크 왕복 시간에 맞춘 `ROUTE_COMMIT_TIMEOUT` 상수로 분리했으며, 만료 시에는 아직 커밋되지 않은 DOM을 상대로 전환을 확정하는 대신 전환을 포기(`skipTransition()`)합니다. 어느 경로로 끝나든 라우팅·히스토리 상태는 정합성을 유지합니다.
- **전환 준비 전에 손을 놓으면 끌어온 거리가 버려지던 문제.** `transition.ready`가 resolve되기 전에 손을 뗀 경우 — 빠른 스와이프이거나 라우트 커밋이 느린 상황 — 애니메이션의 `currentTime`이 0인 채로 재생 또는 되감기가 걸려, 손가락이 멈춘 지점부터 이어지지 않고 처음부터 다시 재생됐습니다. 이제 제스처를 마무리하기 전에 마지막 스크럽 위치를 먼저 적용합니다.
- **플링(튕김) 판정이 전혀 동작하지 않던 문제.** 속도 샘플을 읽기 전에 현재 포인터 위치로 덮어쓰고 있어 비교하는 두 값이 항상 같았고, 계산된 속도가 언제나 0이었습니다. 그래서 짧지만 빠른 스와이프가 취소로 처리됐습니다. 이제 샘플 두 개를 유지하며, 100ms보다 오래된 샘플은 무시해 손을 멈췄다가 뗀 제스처가 지나간 속도로 확정되지 않도록 했습니다.

## 1.0.0

Stack-Link의 화면 전환이 iframe 프리렌더 방식에서 브라우저 네이티브 **View Transitions API** 기반으로 완전히 재구현되었습니다. 전환 엔진 전체를 새로 쓴 변경이며, 1.0 릴리스의 이유이기도 합니다.

### 무엇이 달라졌나

기존에는 모든 전환을 JavaScript가 직접 구동했습니다. iframe이 다음 화면을 미리 렌더링하고, `setTimeout` 체인이 인라인 `transform` 스타일을 단계별로 바꿔가며 애니메이션을 만들었습니다. 이 구조에서는 애니메이션 전체가 메인 스레드 위에 놓이기 때문에, 하이드레이션·데이터 페칭·부하가 걸린 웹뷰처럼 메인 스레드를 함께 쓰는 작업이 있으면 전환 도중 프레임이 밀렸습니다. 어떤 상황에서는 60fps가 나오고 어떤 상황에서는 눈에 띄게 끊기는, 일관성 없는 동작이 되었습니다.

이제는 브라우저가 애니메이션을 직접 관리합니다. 떠나는 화면과 들어오는 화면을 스냅샷으로 캡처해 의사요소로 애니메이션하므로, JavaScript가 무엇을 하고 있든 영향을 받지 않습니다. 전환이 일관되게 부드러워졌고, 웹뷰 환경에서 간헐적으로 발생하던 프레임 드롭이 해소되었습니다.

또한 두 화면이 같은 document 안에 존재하게 되면서 전역 상태·React Context·캐시가 화면 간에 그대로 공유됩니다. iframe 방식은 이동할 때마다 별도 document를 새로 부팅했기 때문에 이 맥락이 모두 끊겼습니다.

### 추가

- **인터랙티브 뒤로가기 제스처.** 엣지에서 스와이프하면 전환 애니메이션이 손가락 위치를 1:1로 실시간 추종합니다. 기존처럼 손을 뗀 뒤 고정된 애니메이션을 재생하지 않습니다. 이동 거리와 튕김 속도를 기준으로 확정 또는 취소를 판단합니다.
- **Pointer Events 지원.** 터치와 마우스 처리를 하나로 통합했으며, `setPointerCapture`로 손가락이 엣지 영역을 벗어나도 제스처가 끊기지 않습니다.
- **모션 최소화 설정 지원.** `prefers-reduced-motion: reduce`가 설정된 환경에서는 전환 애니메이션을 건너뜁니다.
- **자동 폴백.** View Transitions API를 지원하지 않는 브라우저에서는 애니메이션 없이 즉시 이동하며, 오류가 발생하지 않습니다.
- **데모 앱.** 실제 브라우저에서 전환 동작과 화면 간 상태 공유를 확인할 수 있는 예제를 `examples/demo`에 추가했습니다.

### 변경

- **`StackLink`의 `preLoad` 기본값이 `true`로 바뀌었습니다**(기존 `false`). 의미도 함께 달라져, iframe으로 대상 화면을 미리 렌더링하는 대신 `router.prefetch(href)`를 호출합니다. 원하지 않으면 `preLoad={false}`를 명시하세요.
- `useStackLinkRouter`의 `isNavigating`이 훅 내부 ref 대신 Provider가 관리하는 공유 전환 상태를 반영합니다.
- 전환 로직이 `StackLinkProvider`로 이동했습니다. `StackLink`, `useStackLinkBack`, `useStackLinkRouter`는 이를 호출하는 얇은 래퍼가 되었습니다.

### 지원 중단 예정

- **`isInStackFrame`** — iframe 방식에서만 의미가 있던 함수이며, 해당 방식이 사라지면서 더 이상 내부에서 사용하지 않습니다. 하위 호환을 위해 export는 유지하지만 이후 메이저 릴리스에서 제거될 예정입니다.

### 제거

- 내부 `Iframe` 컴포넌트. 공개 API에 포함된 적이 없습니다.
- `#stack-root`, `#stack-previous` DOM 컨테이너와 Provider가 `#stack-main`에 적용하던 인라인 스타일. 전환 스냅샷은 이제 브라우저가 직접 관리합니다.

### 마이그레이션

공개 API는 그대로입니다. `StackLinkProvider`, `StackLink`, `useStackLinkBack`, `useStackLinkRouter` 모두 시그니처가 유지되므로 대부분의 프로젝트는 코드 변경 없이 업그레이드할 수 있습니다.

다만 두 가지는 확인이 필요합니다.

1. `preLoad`가 `false`인 기본 동작에 의존하고 있었다면 값을 명시적으로 지정하세요.
2. 직접 작성한 CSS나 스크립트에서 `#stack-root` 또는 `#stack-previous`를 참조하고 있었다면 해당 규칙을 제거하세요. 이제 존재하지 않는 요소입니다.

### 브라우저 지원

전환 애니메이션은 View Transitions API를 지원하는 브라우저에서 동작합니다. Chrome·Edge 111 이상, Safari 18 이상에서 지원되며, 그 외 브라우저에서는 애니메이션 없이 즉시 이동하는 방식으로 폴백합니다.
